1. Facade Pattern

Objetivo: Fornecer uma interface unificada e simplificada para um subsistema complexo.

No Arbolia: O módulo geminiService.ts atua como uma fachada (Facade). Ele oculta toda a complexidade da API do Google Generative AI (instanciação, injeção da chave da API, construção de prompts formatados) e expõe apenas métodos de alto nível para a aplicação, como getAIInterpretation(). 

// src/services/geminiService.ts
export const getAIInterpretation = async (payload: GeminiRequestPayload) => {
  // Complexidade interna ocultada da UI
  const API_URL = `https://.../gemini-2.0-flash:generateContent?key=${API_KEY}`;
  const response = await fetch(API_URL, { body: buildPrompt(payload) });

  // Componentes React só veem esta função simples
  const data = await response.json();
  return parseResponse(data);
}

2. Singleton Pattern

Objetivo: Garantir que uma classe ou módulo tenha apenas uma instância global na aplicação.

No Arbolia: A inicialização do firebase.ts garante que os serviços (Auth, Firestore, Storage) sejam instanciados uma única vez. Qualquer módulo que importe essas dependências compartilhará a exata mesma instância em memória, evitando duplicação de websockets e garantindo consistência. 

// src/lib/firebase.ts
import { initializeApp } from "firebase/app";

// Inicializa o aplicativo Firebase uma única vez (Singleton)
const app = initializeApp(firebaseConfig);

// Instâncias únicas exportadas globalmente
export const auth = getAuth(app);
export const db = getFirestore(app);
export const storage = getStorage(app);

3. Observer Pattern

Objetivo: Definir uma relação um-para-muitos, onde observadores reagem automaticamente a mudanças de estado.

No Arbolia: Utilizado nativamente na integração Firebase + React. A store do Zustand ou componentes "escutam" ativamente as mudanças de autenticação e de banco de dados (Listeners). Quando o estado muda no Firebase (Subject), a interface atualiza reativamente. 

// Subject (Firebase Auth) emite eventos de mudança de estado
onAuthStateChanged(auth, async (firebaseUser) => {
  if (firebaseUser) {
    // Observers (Zustand/React) reagem para atualizar a UI
    const profile = await fetchUserProfile(firebaseUser.uid);
    set({ user: firebaseUser, userProfile: profile });
  } else {
    set({ user: null, userProfile: null });
  }
});

4. Strategy Pattern

Objetivo: Encapsular famílias de algoritmos para que sejam intercambiáveis sem alterar quem os consome.

No Arbolia: O motor de risco isaRiskEngine.ts encapsula a heurística de cálculo ISA TRAQ. O componente de UI apenas passa os dados (Alvos, Defeitos) e consome o resultado. Se a Arbolia decidir adotar o padrão QTRA no futuro, a UI permanecerá inalterada, bastando injetar uma "Strategy" diferente. 

// Algoritmo de cálculo encapsulado em sua própria estratégia
export function calcularRiscoISA(entradas: EntradaRisco[]): ResultadoISA {
  const matriz1 = calcularMatriz1(entradas);
  const matriz2 = calcularMatriz2(matriz1);
  
  return aplicarMitigacoes(matriz2); // Resultado computado
}

// O componente LaudoModal consome a estratégia sem acoplamento
const resultadoRisco = calcularRiscoISA(entradasConfiguradas);

5. State / Module Pattern

Objetivo: Modularizar e encapsular escopos independentes de estado e comportamento, evitando "God Objects".

No Arbolia: A refatoração recente do Zustand implementou Slices Independentes (authSlice, uiSlice, dataSlice). Em vez de um arquivo massivo, cada módulo governa apenas sua responsabilidade. O useAppStore age apenas como orquestrador e adaptador final para o React. 

// uiSlice.ts: Encapsula puramente regras e mutações de UI
export const createUISlice: AppSlice = (set) => ({
  isSidebarOpen: false,
  toggleSidebar: () => set((s) => ({ isSidebarOpen: !s.isSidebarOpen })),
});

// useAppStore.ts: Composição modular (Store final)
export const useAppStore = create()((...a) => ({
  ...createAuthSlice(...a),
  ...createUISlice(...a),
  ...createDataSlice(...a),
}));
